# Очистка БД от врачей и что нужно «засеять» после

## Скрипт

Файл: `backend/scripts/db_cleanup_doctors.py` (в репозитории)

- По умолчанию **dry-run** (ничего не меняет). Реальное выполнение: `--execute`.
- Перед продакшеном сделайте **бэкап PostgreSQL**.

```bash
cd backend
poetry run python scripts/db_cleanup_doctors.py              # план
poetry run python scripts/db_cleanup_doctors.py --execute --report /tmp/staff.txt
```

Опции:

| Флаг | Действие |
|------|----------|
| `--execute` | Применить изменения |
| `--delete-portal-users` | Дополнительно удалить учётки **только** с ролью `user` (без staff) |
| `--report путь` | Записать отчёт (email staff и роли) |

**Пароли не сбрасываются** — в отчёте только email и роль; вход для staff остаётся прежним, если вы не меняли БД вручную.

**Требование:** в БД должен существовать хотя бы один пользователь с ролью **admin** (иначе скрипт остановится — создайте через `scripts/create_admin.py`).

## Что делает скрипт (логика)

1. Гарантирует строки в таблице **`roles`** (как в `create_admin.py`: admin, manager, accountant, doctor, user).
2. **Нормализует staff:** если у одного пользователя несколько ролей из `admin` / `manager` / `accountant`, остаётся одна по приоритету admin > manager > accountant.
3. **Врачи:** для каждого с ролью `doctor`:
   - если есть staff — снимается роль `doctor`, удаляется `doctor_profile` и связанные данные (голосование по кандидату, content_blocks и т.д.);
   - если staff нет — пользователь удаляется целиком (платежи/квитанции, подписки, ссылки на пользователя в событиях/статьях переназначаются на первого admin).
4. Опционально — удаление «простых» пользователей портала (только роль `user`).

## Нужен ли seed после очистки?

| Данные | Нужно ли заново |
|--------|------------------|
| **`roles`** | Скрипт сам создаёт недостающие роли. |
| **`plans`** (тарифы: вступительный, годовой и т.д.) | **Не удаляются** скриптом. Если удаляли БД целиком — нужны начальные планы (миграции/ручной ввод/фикстуры). |
| **`cities`** | Скрипт не трогает. Пустой справочник — добавить через админку или SQL. |
| **`site_settings`** | Не трогаются. |
| **`notification_templates`** | Не трогаются. |
| **Первый admin** | Если после полного сброса БД никого нет — `python scripts/create_admin.py`. |
| **Импорт врачей / Excel** | Отдельный пайплайн (см. `docs/seed/MIGRATION_PARSER_REQUIREMENTS.md`) — не часть этого скрипта. |

Итого: после **частичной** очистки (только скрипт) справочники и тарифы обычно **остаются**. После **полного пересоздания** БД нужны миграции `alembic upgrade head`, `create_admin`, при необходимости начальные **plans** и **города** — как при первом деплое.

## Ограничения и риски

- Ссылки `created_by` / `author_id` / `created_by` у голосований на удалённых врачей переносятся на **первого попавшегося admin** — историческая точность авторства теряется.
- Удаление необратимо (кроме восстановления из бэкапа).
