# Техническое задание

## Доработка: Протокольный workflow выдачи сертификата и статус «Кандидат в члены»

**Проект:** trichologia.ru | **Дата:** 02.04.2026 | **Версия:** 1.1

---

## 1. Описание задачи

Заказчик требует изменить порядок активации членства и выдачи сертификатов. Сейчас после оплаты врач сразу получает статус «active» и сертификат. Нужно добавить промежуточный статус — «Кандидат в члены» — между оплатой и полным членством.

**Ключевое изменение v1.1:** при успешной оплате система автоматически создаёт запись в истории протокола со статусом «Ожидает подтверждения». Бухгалтер/админ в админке подтверждает или отклоняет эту запись. При подтверждении — генерируется сертификат и врач получает уведомление (email + Telegram). При отклонении — сертификат не выдаётся, врач получает уведомление об отказе.

---

## 2. Для заказчика: как это будет работать

> Этот раздел описывает процесс простым языком — без технических деталей. Здесь показан путь врача и путь администратора/бухгалтера от начала до конца.

### Путь врача (клиентский сайт)

1. **Врач оплачивает членский взнос** на сайте через форму оплаты.
2. После оплаты в личном кабинете появляется статус **«Кандидат в члены»** и сообщение: *«Оплата получена. Ожидайте подтверждения. Заседания правления проводятся ежеквартально.»*
3. Врач уже виден в каталоге специалистов и получает скидки на мероприятия, но сертификат пока не выдан.
4. Когда бухгалтер/админ подтверждает приём — врач получает **уведомление на почту** и **в Telegram** (если привязан): *«Поздравляем! Вы приняты в ассоциацию.»*
5. В личном кабинете статус меняется на **«Член ассоциации»**, и врач может **скачать сертификат**.
6. Если заявка отклонена — врач получает уведомление с причиной отказа. Сертификат не выдаётся.

### Путь администратора / бухгалтера (админка)

1. Врач оплачивает взнос — **система автоматически** создаёт запись в разделе «История протокола» со статусом **«Ожидает подтверждения»**.
2. Админ/бухгалтер заходит в раздел **«История протокола»** — видит список записей, среди которых есть новые со статусом «Ожидает подтверждения» (выделены жёлтым).
3. Админ открывает запись и принимает решение:
   - **«Подтвердить»** — система автоматически: активирует членство, генерирует PDF-сертификат, отправляет врачу уведомление на почту и в Telegram.
   - **«Отклонить»** (с указанием причины) — сертификат не генерируется, врач получает уведомление об отказе.
4. Никаких дополнительных действий не требуется — всё происходит автоматически после нажатия кнопки.

### Таблица: что меняется (до / после)

| Этап | Сейчас (до) | Будет (после) |
|------|-------------|---------------|
| Оплата взноса | Подписка сразу активна, сертификат генерируется автоматически | Подписка в статусе «Кандидат», в истории протокола автоматически появляется запись «Ожидает подтверждения» |
| Запись в протокол | Админ вручную заносит запись, она ни на что не влияет | Запись создаётся автоматически при оплате. Админ только подтверждает или отклоняет |
| Выдача сертификата | Автоматически при оплате | Только после подтверждения записи в протоколе бухгалтером/админом |
| Уведомление врача | Только об оплате | Дополнительно: о принятии в ассоциацию (email + Telegram) или об отклонении |
| Исключение из ассоциации | Нет автоматизации | Автоматический протокол исключения при неуплате 2+ года |
| Действия админа | Вручную заносить протокол + вручную управлять статусами | Только подтвердить/отклонить — всё остальное автоматически |

Добавить значение `candidate` в PostgreSQL enum `SubscriptionStatus`. После успешной оплаты подписка переходит в `candidate` (вместо `active`). Привилегии каталога и цен мероприятий работают для `candidate` так же, как для `active`.

### 5.2 Новое поле в модели ProtocolHistoryEntry: confirmation_status

Добавить поле `confirmation_status` в модель `ProtocolHistoryEntry` с enum-значениями:

| Значение | Описание |
|----------|----------|
| `pending_confirmation` | Запись создана автоматически при оплате, ожидает решения бухгалтера/админа |
| `confirmed` | Бухгалтер подтвердил — сертификат выдан |
| `rejected` | Бухгалтер отклонил — сертификат не выдаётся |

Дополнительные поля:
- `confirmation_status` — `String(30)`, default `pending_confirmation`
- `confirmed_by_user_id` — `ForeignKey("users.id")`, nullable — кто подтвердил/отклонил
- `confirmed_at` — `DateTime`, nullable — когда подтвердили/отклонили
- `rejection_reason` — `Text`, nullable — причина отклонения
- `auto_generated` — `Boolean`, default `false` — флаг автоматически созданной записи (при оплате)

**Файлы:**
- `backend/app/models/protocol_history.py` — добавить поля
- `backend/app/schemas/protocol_history.py` — обновить схемы
- Alembic миграция — `ALTER TABLE protocol_history_entries ADD COLUMN ...`

**CheckConstraint:**
```sql
CHECK (confirmation_status IN ('pending_confirmation', 'confirmed', 'rejected'))
```

### 5.3 Автоматическая генерация записи протокола при оплате

При успешной оплате (`entry_fee` / `subscription`) — webhook-сервис автоматически создаёт запись в `protocol_history_entries`:

```python
# В _apply_payment_succeeded(), после активации подписки в candidate:
entry = ProtocolHistoryEntry(
    year=now.year,
    protocol_title=f"Заявка на вступление — оплата {now.strftime('%d.%m.%Y')}",
    notes=None,
    doctor_user_id=payment.user_id,
    action_type="admission",
    confirmation_status="pending_confirmation",
    auto_generated=True,
    created_by_user_id=payment.user_id,  # автор — сам врач (авто)
)
db.add(entry)
```

**Файлы:**

### 5.5 Подтверждение / отклонение записи протокола (админ API)

Новые эндпоинты в `protocol_history_admin.py`:

**`PATCH /admin/protocol-history/{entry_id}/confirm`**
- Роли: `admin`, `manager`, `accountant`
- Логика:
  1. Проверить `confirmation_status == pending_confirmation`
  2. Установить `confirmation_status = confirmed`, `confirmed_by_user_id`, `confirmed_at`
  3. Найти подписку врача в статусе `candidate` → перевести в `active`
  4. Запустить фоновую задачу генерации сертификата (`generate_member_certificate_task`)
  5. Отправить уведомление врачу (email + Telegram): «Вы приняты в ассоциацию»
- Ответ: обновлённая `ProtocolHistoryResponse`

**`PATCH /admin/protocol-history/{entry_id}/reject`**
- Тело: `{ "rejection_reason": "Причина отказа" }` (обязательное поле)
- Логика:
  1. Проверить `confirmation_status == pending_confirmation`
  2. Установить `confirmation_status = rejected`, `confirmed_by_user_id`, `confirmed_at`, `rejection_reason`
  3. Подписка остаётся `candidate` (сертификат НЕ генерируется)
  4. Отправить уведомление врачу (email + Telegram): «Заявка отклонена» + причина
- Ответ: обновлённая `ProtocolHistoryResponse`

**Файлы:**
- `backend/app/api/v1/protocol_history_admin.py` — новые эндпоинты
- `backend/app/services/protocol_history_admin_service.py` — методы `confirm()` и `reject()`
- `backend/app/schemas/protocol_history.py` — схемы запроса/ответа для confirm/reject

### 5.7 Сертификат: генерация только после подтверждения протокола

Убрать автоматическую генерацию сертификата из webhook-сервиса. Сертификат генерируется ТОЛЬКО при подтверждении записи протокола (метод `confirm()` в `ProtocolHistoryAdminService`).

**Логика подтверждения (атомарная транзакция):**
1. `confirmation_status` → `confirmed`

4. Уведомление email + Telegram

**Файлы:**

### 5.8 API: сообщения в списке сертификатов

Endpoint `GET /certificates` должен возвращать информационное сообщение в зависимости от состояния врача:

| Состояние | Сообщение в ответе API (поле `message`) |
|----------|----------------------------------------|
| Нет подписки / не оплатил | «Для получения сертификата необходимо оплатить членский взнос и дождаться подтверждения.» |
| Подписка = `candidate`, протокол = `pending_confirmation` | «Оплата получена. Ваша заявка ожидает подтверждения. Заседания правления проводятся ежеквартально.» |
| Подписка = `candidate`, протокол = `rejected` | «Ваша заявка отклонена. Причина: {rejection_reason}. Свяжитесь с нами для уточнений.» |
| Подписка = `active` | Без сообщения, выводится список сертификатов |

### 5.9 Автоматический протокол исключения (2 года неуплаты)

Добавить scheduled job (TaskIQ / cron): раз в день проверять врачей с expired подпиской > 730 дней. Для каждого автоматически создавать `ProtocolHistoryEntry` с `action_type = "exclusion"`, `confirmation_status = "confirmed"`, `auto_generated = true` и деактивировать профиль. Сертификаты деактивируются (`is_active = false`).

### 5.10 Миграция базы данных

1. `ALTER TYPE subscriptionstatus ADD VALUE 'candidate'`
2. `ALTER TABLE protocol_history_entries ADD COLUMN confirmation_status VARCHAR(30) DEFAULT 'confirmed' NOT NULL` — для обратной совместимости: все существующие записи = `confirmed`
3. `ALTER TABLE protocol_history_entries ADD COLUMN confirmed_by_user_id UUID REFERENCES users(id)`
4. `ALTER TABLE protocol_history_entries ADD COLUMN confirmed_at TIMESTAMPTZ`
5. `ALTER TABLE protocol_history_entries ADD COLUMN rejection_reason TEXT`
6. `ALTER TABLE protocol_history_entries ADD COLUMN auto_generated BOOLEAN DEFAULT false NOT NULL`
7. Существующие подписки НЕ меняются (обратная совместимость)

## 11. Риски и что учесть

- **Обратная совместимость:** существующие `active` подписки не должны сломаться. Миграция добавляет новое значение enum и новые поля с default-значениями — существующие строки не затрагиваются. Все старые записи протокола получат `confirmation_status = confirmed`.
- **Фронтенды** должны обрабатывать новый статус `candidate` и новые поля `confirmation_status`, `rejection_reason`. До обновления фронта — graceful degradation.
- **Scheduler автоисключения** должен запускаться только если включена настройка (`site_settings`), чтобы можно было отключить.
- **Генерация сертификата** при подтверждении протокола — обновление статуса подписки и запуск задачи генерации в одной транзакции; PDF генерируется в фоновой задаче.
- **Уведомления при отклонении** — причина отказа обязательна (`rejection_reason`), чтобы врач понимал почему отказали.
- **Дубли автогенерации** — при повторной оплате проверять, нет ли уже записи протокола `pending_confirmation` для данного врача и года, чтобы не создавать дубли.
