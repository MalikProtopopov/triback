# План и требования к парсеру и итоговой таблице миграции

Документ задаёт ТЗ на пайплайн сбора данных для переноса с [trichologia.ru](https://trichologia.ru/) и объединения с Excel (`Копия Членство ассоциации.xlsx`, вкладка **«текущие»**). Используется при планировании реализации (Scrapy, Python, Poetry, Docker) и последующей загрузки в БД бэкенда этого репозитория.

---

## Контекст

Нужно спроектировать **пайплайн сбора данных** для переноса контента с действующего сайта и объединения с **Excel** (`docs/seed/Копия Членство ассоциации.xlsx`). Вкладка **«текущие»** — только врачи с активной подпиской по мнению ассоциации. Даты начала подписки в источниках **неизвестны**; допустима политика **«действует до 31.12 текущего календарного года»** для импорта, с явной пометкой в таблице.

Целевая система — **бэкенд этого репозитория**: пользователи, `doctor_profiles` (в т.ч. `status`, `slug`, `photo_url`), подписки (`subscriptions`: `plan_id`, `status`, `starts_at`, `ends_at`), платежи (`payments`: `payment_provider`, в т.ч. ручной/импорт).

**Стек реализации парсера (обязательно зафиксировать в плане):** Python, **Poetry** для зависимостей, **Scrapy** для обхода сайта, **Docker** (образ для воспроизводимого запуска краулера и постобработки).

---

## Цели

1. **Единая нормализованная таблица** (CSV/Excel + при необходимости JSON sidecar), по строке на одного врача (или одну логическую сущность «член + профиль»), пригодная для **ручной доработки** и последующей **загрузки в БД**.
2. **Парсер сайта**, который извлекает максимум полей с публичных страниц (список врачей, карточки, контакты, тексты, ссылки на фото).
3. **Слияние** с Excel: сопоставление по ФИО, email, телефону, slug/URL — с колонками **источник значения** и **уверенность матчинга**.
4. **Медиа:** скачать изображения в отдельную папку (`assets/doctors/` или аналог), в таблице — **стабильное имя файла** или **ключ** + относительный путь.
5. **Бизнес-правила из Excel:**
   - **Зелёная заливка строки** = подписка **истекла**, требуется **продление** (в таблице — отдельный флаг/статус импорта подписки).
   - **Активные** (из вкладки «текущие» и не зелёные по правилам) — должны иметь в целевой модели **активную подписку** и **публичный профиль**; если не хватает обязательных полей для публикации — профиль в статусе **на модерации** (`pending_review` / черновик изменений — согласовать с enum `DoctorStatus` в бэкенде: `pending_review`, `approved`, `active`, …).
6. **Платежи при импорте:** заложить в требования явное поле **«источник платежа»** = например `import_script` / `manual_migration` / `payment_provider=manual` с **описанием** в `description`, чтобы в БД было видно, что подписка выставлена миграцией, а не реальным эквайрингом.
7. **Пустые поля:** любая ячейка, которую нельзя заполнить из сайта или Excel, должна оставаться пустой, но рядом (или в отдельных колонках) — **`needs_manual_review`**, **`missing_fields`**, **`notes`**.

---

## Нефункциональные требования

- Уважать **robots.txt** и нагрузку на сайт: задержки, лимит concurrency в Scrapy, идентифицируемый User-Agent.
- **Идемпотентность:** повторный запуск не ломает папку с картинками (перезапись по хешу или по детерминированному имени).
- **Воспроизводимость:** версии зависимостей в `pyproject.toml` (Poetry), Dockerfile с фиксированным базовым образом.
- **Логи:** отдельный лог матчинга (кто с кем сопоставлен, почему doubtful).

## Колонки итоговой таблицы (минимальный набор — расширить в плане)

| Колонка | Описание |
|---------|----------|

### Пользователь / профиль (маппинг на `users` + `doctor_profiles`)

| Колонка | Описание |
|---------|----------|

| `last_name`, `first_name`, `middle_name` | ФИО |
| `city_name` | При импорте в БД — `city_id` отдельным справочником |

| `profile_visibility_note` | Например «публичный после заполнения полей X» |

| Колонка | Описание |
|---------|----------|

## Требования к парсеру (Scrapy)

1. **Spider(ы):** старт с раздела «Врачи» / каталога (точные URL уточнить при разведке и зафиксировать в плане).
2. **Извлечение:** ФИО, должность, клиника, город, био, все изображения профиля (оригинал по возможности), контакты если есть.
3. **Pipeline:** сохранение сырого HTML/JSON на строку (опционально для отладки), загрузка файлов, запись строки в промежуточный CSV.
4. **Обход ссылок:** только в пределах домена и разрешённых путей.
5. **Выход:** `raw_site_export.csv` + папка `assets/`.

---

## Этап слияния (отдельный модуль или скрипт)

- Читает Excel (openpyxl/pandas) с учётом **цвета ячейки/строки** для флага «зелёный = истекла».
- Нормализация ФИО (ё/е, дефисы, пробелы).
- Join с `raw_site_export` по правилам из раздела матчинга.
- Производит **master_table.csv** / **master_table.xlsx**.

---

## Критерии приёмки

- Есть Docker-инструкция: `docker compose run crawler` (или аналог) → артефакты на диске.
- Мастер-таблица открывается в Excel, все колонки из раздела «качество» заполняются алгоритмически где возможно.
- Для каждой строки с неполными данными — понятно, **что** добить вручную.
- Документирована политика дат подписки (календарный год) и пометка в `payment_description`.

## Фазы работ (ориентир)

1. Разведка сайта (URL, структура HTML).
2. Схема итоговой таблицы и контракт с БД.
3. Scrapy-проект + выгрузка `raw_site_export.csv` и `assets/`.
4. Merge с Excel → `master_table`.
5. Валидация и отчёт по `needs_manual_review`.
6. Отдельный скрипт импорта в БД (вне этого документа — по готовности мастер-таблицы).
