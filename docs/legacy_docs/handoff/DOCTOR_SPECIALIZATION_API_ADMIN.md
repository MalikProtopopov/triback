# API: специализация врача (админка)

Специализация врача — **строка до 255 символов** в таблице профиля, без справочника `specializations` и без `specialization_ids` в JSON.

## Создание врача

| Метод | Путь |
|--------|------|
| POST | `/api/v1/admin/doctors` |

**Тело (фрагмент):**

- `specialization`: `string \| null`, опционально, max 255.

**Удалено:** `specialization_ids` (массив UUID) — больше не принимается.

## Список и карточка врача (модерация)

| Метод | Путь |
|--------|------|
| GET | `/api/v1/admin/doctors` |
| GET | `/api/v1/admin/doctors/{profile_id}` |

В каждой записи / в деталке: `specialization: string \| null` — текст из профиля.

## Портальные пользователи

| Метод | Путь |
|--------|------|
| GET | `/api/v1/admin/portal-users` |
| GET | `/api/v1/admin/portal-users/{user_id}` |

Для пользователей с ролью врача и существующим профилем: поле `specialization: string \| null`. Для пользователей без профиля врача — `null`.

## Важно: правка после создания

Отдельного **PATCH** профиля врача в админке (в т.ч. только для смены специализации) **нет**. После `POST /admin/doctors` изменить специализацию через админский API нельзя — только прямое изменение в БД или будущий эндпоинт правки профиля.

Пользователь-врач может менять специализацию из ЛК (`/profile/personal`, черновик `/profile/public`) в рамках клиентского API — см. [DOCTOR_SPECIALIZATION_API_CLIENT.md](./DOCTOR_SPECIALIZATION_API_CLIENT.md).

## Миграция данных

При применении миграции на проде:

1. Значение из старого `specialization_id` (название из справочника) переносится в текстовое поле.
2. Если было несколько специализаций через M:N, в текст записывается объединение имён через запятую (сортировка по имени).

## Фронтенд админки

- Форма создания врача: одно текстовое поле вместо мультивыбора по UUID.
- Список/карточка: отображать `specialization` как строку.
