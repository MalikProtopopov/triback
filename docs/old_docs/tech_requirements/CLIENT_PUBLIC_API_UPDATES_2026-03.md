# Обновления публичного API — клиентский фронтенд

**Дата:** 2026-03-17  
**Релиз:** Модерация публичных данных, фильтр по подписке

---

## Кратко

На бэкенде изменена логика показа врачей и загрузки фото. Клиентскому сайту нужно учесть:

1. **Врачи без активной подписки** больше не возвращаются в публичном API
2. **Фото профиля** у врача загружаются через модерацию — изменения применяются только после одобрения админом
3. **Подсчёт врачей в городах** учитывает только врачей с активной подпиской

---

## 1. Публичные эндпоинты врачей

### `GET /api/v1/doctors`

**Изменение:** В выдачу попадают только врачи с:
- `status = "active"`
- Действующей подпиской (`Subscription.status = "active"`, `ends_at > now` или `ends_at = null`)

**Для фронтенда:** Логика запроса и параметров (`city_id`, `city_slug`, `specialization`, `search`) не меняется. Меньше врачей в ответе — это ожидаемо (истекшая подписка = врач скрыт с сайта).

### `GET /api/v1/doctors/{identifier}`

**Изменение:** Возвращает `404 Not Found`, если у врача нет активной подписки.

**Для фронтенда:**
- Если раньше врач был доступен по slug/UUID, а теперь 404 — подписка истекла
- Обрабатывать 404 как «врач не найден» или «недоступен»
- Не закешировать старые данные; при 404 — показывать страницу 404 или «Врач временно недоступен»

### `GET /api/v1/cities?with_doctors=true`

**Изменение:** В `doctors_count` входят только врачи с активной подпиской.

**Для фронтенда:** Поле `doctors_count` может уменьшиться; изменения в схеме нет.

### `GET /api/v1/cities/{slug}`

**Изменение:** `doctors_count` также учитывает только врачей с активной подпиской.

## 2. Личный кабинет врача (авторизованные запросы)

### `POST /api/v1/profile/photo`

**Изменение:** Фото больше не обновляется сразу. Оно уходит в черновик и публикуется только после одобрения админом.

**Новый формат ответа:**

```json
{
  "photo_url": "https://api.trichologia.ru/media/doctors/{uuid}/photo/xxx.jpg",
  "message": "Фото загружено и отправлено на модерацию",
  "pending_moderation": true
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `photo_url` | string | URL загруженного фото (для превью) |
| `message` | string | Сообщение пользователю |
| `pending_moderation` | boolean | **Новое.** Всегда `true` после загрузки — изменение на модерации |

**Для фронтенда:**
1. Показывать превью нового фото по `photo_url`
2. Показывать предупреждение: «Фото отправлено на модерацию. Изменения появятся на сайте после одобрения администратором»
3. Использовать `pending_moderation`, чтобы отличать «фото ещё не применено» от «старое фото, уже опубликовано»

### `GET /api/v1/profile/public`

**Изменение:** В `pending_draft.changes` может появиться поле `photo_url` (S3-ключ).

**Пример при ожидающем фото:**

```json
{
  "bio": "...",
  "photo_url": "https://api.trichologia.ru/media/old.jpg",
  "pending_draft": {
    "status": "pending",
    "changes": {
      "photo_url": "doctors/uuid/photo/new-key.jpg"
    },
    "submitted_at": "2026-03-17T12:00:00Z"
  }
}
```

`photo_url` в корне — текущее одобренное фото.  
`pending_draft.changes.photo_url` — S3-ключ нового фото; для отображения нужно добавить базовый URL медиа, например:  
`${MEDIA_BASE_URL}/${changes.photo_url}`.

**Для фронтенда:**
1. Если есть `pending_draft` с `photo_url` в `changes` — показывать превью нового фото
2. Подписать: «Новое фото на модерации»
3. Текущее `photo_url` в корне — то, что сейчас на сайте

## 3. Поведение при ошибках

| Ситуация | Раньше | Сейчас |
|----------|--------|--------|
| Врач без подписки | В списке / по slug | Не в списке, 404 по slug |
| Врач с истекшей подпиской | В списке / по slug | Не в списке, 404 по slug |
| Загрузка фото | Фото сразу в профиле | Фото в черновике, ждёт одобрения |

---

## 4. Рекомендации по реализации

1. **Страница врача** — при 404 от `GET /doctors/{slug}` показывать «Врач временно недоступен» или страницу 404
2. **Страница «Каталог врачей»** — при пустом списке проверять, есть ли фильтры; учитывать, что часть врачей может быть скрыта из‑за подписки
3. **Загрузка фото в ЛК** — показывать индикатор «На модерации» (`pending_moderation`) и не считать фото уже обновлённым на сайте
4. **Превью фото в черновике** — собирать URL из `MEDIA_BASE_URL + pending_draft.changes.photo_url`, если `photo_url` присутствует в `changes`
