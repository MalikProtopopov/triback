# Админка: обновления API (hot-fix серии)

Документ описывает изменения в бэкенд-API, которые надо отразить на фронтенде админки. Все правки — **аддитивные и обратно-совместимые**: существующие вызовы продолжат работать без изменений, но не покажут новых данных/возможностей.

## 1. `/admin/payments` — фильтр по ФИО / email

### Новый query-параметр

| Параметр | Тип | Описание |
|---|---|---|
| `name` | `string` (optional) | Частичный поиск по ФИО пользователя (поля `last_name`, `first_name`, `middle_name` из `doctor_profiles`) и по `email`. |

**Семантика поиска.** Строка разбивается по пробелам на токены. Каждый токен должен быть найден хотя бы в одном из указанных полей (ILIKE `%token%`). Примеры:

- `?name=Романова` → все платежи, где фамилия ИЛИ имя ИЛИ отчество ИЛИ email содержит «Романова».
- `?name=Романова Юлия` → платежи, где один токен попал в ФИО/email, а другой — ещё в какое-то поле ФИО/email того же пользователя. То есть «Романова Юлия» и «Юлия Романова» оба матчатся.
- `?name=opueva@gmail` → матч по подстроке email.

### UI-рекомендация

Добавить одно текстовое поле «ФИО / email» над таблицей платежей. Debounce 300–500 мс перед отправкой запроса.

## 2. `/admin/payments` — фильтр по внешнему ID провайдера

### Новый query-параметр

| Параметр | Тип | Описание |
|---|---|---|
| `provider_id` | `string` (optional) | Точное совпадение с `external_payment_id` **или** `moneta_operation_id`. |

Пример запроса:
```
GET /admin/payments?provider_id=123456789
```

### UI-рекомендация

Добавить отдельное поле «ID в платёжной системе» рядом с поиском по ФИО. Без debounce — триггер по Enter или кнопке.

## 3. `/admin/payments` — новые поля в ответе

В схему `PaymentListItem` (элементы массива `data`) добавлены два поля:

| Поле | Тип | Пусто когда |
|---|---|---|
| `external_payment_id` | `string | null` | Для ручных платежей (provider = `manual`) или если провайдер ещё не вернул ID. |
| `moneta_operation_id` | `string | null` | Всегда `null` для YooKassa-платежей; заполнено только для `payment_provider = "moneta"`. |

Пример ответа:
```json
{
  "data": [
    {
      "id": "2da5c87d-000f-5000-8000-1a2b3c4d5e6f",
      "user": { "id": "…", "email": "user@example.com", "full_name": "Романова Юлия" },
      "amount": 5000.0,
      "product_type": "subscription",
      "payment_provider": "moneta",
      "status": "succeeded",
      "external_payment_id": "987654321",
      "moneta_operation_id": "987654321",
      "paid_at": "2026-04-16T12:34:56+00:00",
      "created_at": "2026-04-16T12:34:50+00:00"
    }
  ],
  "summary": { "total_amount": 5000.0, "count_completed": 1, "count_pending": 0 },
  "total": 1,
  "limit": 20,
  "offset": 0
}
```

### UI-рекомендация

Показать оба ID в строке платежа (например, под статусом или в popover-карточке), чтобы сотрудник мог скопировать и сверить с ЛК «Монеты». Рекомендуется формат `#<short_id>` + кнопка «копировать».

## 4. Фильтр по датам — `date_from` / `date_to`

### Что изменилось (важно)

Раньше: запрос `?date_from=2026-04-16&date_to=2026-04-16` возвращал `0` платежей, потому что сравнение шло с полуночью (`<= 2026-04-16T00:00:00`).

Теперь:
- Если клиент прислал **календарную дату** (без времени либо с временем `00:00:00`), бэк трактует её как MSK-день (`Europe/Moscow`) и расширяет `date_to` до конца дня.
- Если клиент прислал **полный ISO datetime** (например, `2026-04-16T15:30:00Z`), бэк использует его как есть.

### Действия на фронте

- **Предпочтительный вариант.** В date-picker брать только дату (`YYYY-MM-DD`) и отправлять её без времени. Бэк сам развернёт в MSK-день.
- **Альтернатива**, если дёргаете `Date` в JS и он сериализуется как ISO: убедитесь, что время = `00:00:00` в UTC, либо не отправляйте время вовсе.
- **Не** добавляйте на фронте хак «пиши `T23:59:59` в `date_to`». Это больше не нужно — и если вдруг было сделано, уберите.

Применяется также к `/admin/events` (те же параметры `date_from` / `date_to`).

## 5. Восстановление пароля

Не влияет на UI — исправление инфраструктурное (case-insensitive поиск пользователя в `/auth/forgot-password`). Пользователю, который ранее не мог восстановить пароль из-за другого регистра email, теперь придёт письмо.

**Единственное изменение на фронте:** при вводе email в формах регистрации/входа/восстановления пароля рекомендуется делать `email.trim().toLowerCase()` перед отправкой — чтобы UI и бэк видели одинаковую строку. Бэк всё равно нормализует сам, но это поможет избежать визуальных расхождений (например, email в уведомлениях из формы vs из БД).
