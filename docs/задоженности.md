## Архитектурно (чтобы не разъехалось с тестами и поддержкой)

Имеет смысл **не** плодить десятки разовых `Payment` без связи, а ввести явную модель обязательств, например:

- **`MembershipYearObligation`** (или аналог): `user_id`, **календарный год**, сумма, статус `open | paid | waived | cancelled`, источник `system | admin`, опционально ссылка на оплативший `payment_id`.
- Очередь оплаты для ЛК: API **собирает корзину**: сначала все открытые долги по годам (и при исключении — вступительный + долги), потом — год текущий/следующий, как вы зафиксируете в правилах.
- **`ProductType`**: либо новое значение вроде **`membership_arrears`** (и в платеже **metadata/отдельная строка** с годом), либо отдельная таблица line-items к одному чеку — важно, чтобы webhook при success **закрывал** нужные obligations по порядку.
- **Флаги на пользователе** (или профиле врача): например `entry_fee_exempt` / `force_entry_fee` — чтобы миграция и админ не ломали правило «по умолчанию все платят вступительный, кроме явно помеченных».

Параллельно придётся **пересмотреть**:

1. **`determine_product_type` и `pay()`** — сейчас они не знают про долги и про «сначала погасить долг».
2. **`_activate_subscription`** — если нужна привязка «один платёж закрыл N лет долга + продлил подписку до 31.12», это отдельная транзакционная логика.
3. **Исключение через 3 года** — отдельное состояние пользователя/членства (не только `subscription.expired`), иначе смешается с обычным просроченным годом.
